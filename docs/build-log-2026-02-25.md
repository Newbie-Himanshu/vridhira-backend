## Overview

This session delivered a **GA4 Analytics admin dashboard** — a full server-side Google Analytics 4 reporting page inside the Medusa admin panel. The implementation purposefully avoids gRPC and native bindings so it works reliably on Windows dev machines and in any Node environment without extra compilation steps.

---

## Features Implemented

### 1. GA4 Data API Backend Route

**File:** `src/api/admin/custom/ga4/route.ts`
**Endpoint:** `GET /admin/custom/ga4?days=30`

The route is a thin REST proxy that:

1. Checks for required env vars (`GA_PROPERTY_ID`, `GA_SERVICE_ACCOUNT_KEY`). Returns a structured `{ configured: false, missing: [...] }` response if either is absent — the frontend uses this to render a setup guide instead of crashing.
2. Exchanges a signed RS256 JWT for a short-lived Google OAuth2 access token (token is cached in memory for 55 min to avoid hammering the token endpoint).
3. Fires four parallel GA4 Data API requests via `fetch`:
   - **Summary metrics** — sessions, activeUsers, screenPageViews, bounceRate, averageSessionDuration, newUsers
   - **Event breakdown** — top 25 events by count (split into ecommerce events + all events)
   - **Top pages** — top 10 pages by views
   - **Daily trend** — sessions + users per day for the selected range
4. Surfaces GA4 API errors gracefully: if the Data API isn't enabled in Google Cloud, the response includes `activation_url` extracted from the error's `details[].metadata.activationUrl` field so the frontend can render a direct "Enable API" button.

**Why plain HTTPS instead of the `@google-analytics/data` npm package?**

The `@google-analytics/data` package uses gRPC under the hood, which requires native bindings (`grpc_node.node`) that frequently fail to compile on Windows. The plain HTTPS approach uses Node's built-in `crypto.createSign` for JWT signing and `fetch` for HTTP — zero extra dependencies, zero compilation.

**Authentication flow:**

```
Service Account JSON key
  → RS256-signed JWT (header.payload.sig, base64url)
  → POST https://oauth2.googleapis.com/token
  → { access_token, expires_in }
  → Bearer token on GA4 Data API calls
```

**Required env vars:**

```
GA_PROPERTY_ID=503245289            # numeric ID from GA4 admin URL (/pXXXXX/admin)
GA_SERVICE_ACCOUNT_KEY={"type":"service_account","client_email":"...","private_key":"..."}
```

**Service account setup:**
1. Google Cloud Console → IAM → Service Accounts → Create
2. Create JSON key → download
3. GA4 Admin → Property Access Management → Add service account email as **Viewer**
4. Paste the entire JSON file content (as a single line) into `GA_SERVICE_ACCOUNT_KEY`

### 2. GA4 Analytics Admin Page

**File:** `src/admin/routes/ga4/page.tsx`
**Route:** `/app/ga4` (sidebar label: "GA4 Analytics", icon: ChartBar)

A full-featured analytics dashboard rendered inside the Medusa admin panel.

#### States

| State | Condition | UI |
|-------|-----------|-----|
| Loading | Query in flight | Spinner + "Loading GA4 data…" |
| Fetch error | Network/500 from backend | Error message + Retry button |
| Not configured | `configured: false` | Setup guide listing missing env vars |
| API error | `configured: true`, `error` set | Error message + hint + Enable/Open GA4 buttons |
| Dashboard | `configured: true`, no error | Full metrics UI |

#### Dashboard sections

**Header**
- Property ID badge and server-side tracking status (green/orange)
- Day-range toggle: 7 / 30 / 90 days (re-fetches on change, 5-min stale time)
- "Open GA4" button → GA4 admin for this property

**Metric cards (6)**
Sessions · Active Users · New Users · Page Views · Bounce Rate · Avg. Session Duration

**Two-column grid**
- *Ecommerce Events* — server-side tracked events with friendly labels (Add to Cart, Purchase, etc.)
- *Top Pages* — path + view count, top 10

**Top Events grid**
All events (not just ecommerce), displayed as compact cards with monospace event names

**Daily Trend table**
Last 14 rows of the selected date range (most recent first), columns: Date · Sessions · Users

#### Notable implementation details

- Uses `@tanstack/react-query` `useQuery` with `retry: false` so GA4 errors surface immediately rather than retrying three times
- Number formatting: K / M suffixes for large counts (`fmt()`), `Xm Ys` for durations (`fmtDuration()`)
- Day-range filter is implemented as a custom segmented button (not a Select) to match the compact admin UI style
- `GA_MEASUREMENT_ID` presence is checked separately and shown as a badge — lets you see at a glance whether server-side event tracking is also active

# Session 2 — February 25, 2026 (GA4 Multi-Tab Expansion)

## Overview

Extended the single-page GA4 dashboard into a full **5-tab analytics suite** with dedicated backend API routes and rich frontend pages. Added **GA4 hotlink buttons** to every admin widget for one-click navigation to the relevant analytics tab.

---

## Architecture

### Shared backend module — `src/api/admin/custom/ga4/_lib.ts`

Extracted all duplicated auth and caching logic into a single shared module used by all 5 API routes:

- **`getAccessToken(key)`** — Signs RS256 JWT, exchanges for OAuth2 Bearer token, caches for 55 min
- **`runReport(token, propertyId, body)`** — Calls GA4 Data API REST endpoint, parses errors, throws structured objects
- **`getCachedReport(key, ttl, fn)`** — Generic 15-min TTL cache using a `Map<string, {data, expiresAt}>` singleton
- **`resolveGAConfig()`** — Reads + validates env vars; returns `{propertyId, key}` or throws with missing var names

### Shared frontend library — `src/admin/lib/ga4-shared.tsx`

Single import point for all GA4 pages:

| Export | Purpose |
|--------|---------|
| `MetricCard` | KPI card with label, value, optional sub-label |
| `DaysFilter` | 7 / 30 / 90 day segmented button |
| `GA4NavCard` | Navigation card with icon, title, description, hotkey badge |
| `GA4PageHeader` | Standard page header with property badge, day filter, open-GA4 button |
| `LoadingState` / `ErrorState` / `APIErrorState` | Consistent loading + error UIs |
| `useGA4Hotkeys()` | Registers G→A→* keyboard shortcuts via `useEffect` |
| `HOTKEYS` | Hotkey map used for badges and handler logic |
| `inr()` | Format number as ₹ with K/L/Cr suffixes |
| `fmt()` | Format number with K/M suffixes |
| `fmtDuration()` / `fmtPct()` | Duration (Xm Ys) and percentage formatting |
| `sdk` | Medusa SDK client singleton |

## New API Routes

### Performance — `GET /admin/custom/ga4/performance?days=N`

**File:** `src/api/admin/custom/ga4/performance/route.ts`

Two parallel GA4 reports:
1. **KPI summary** — `purchaseRevenue`, `transactions`, `averagePurchaseRevenue`, `sessionConversionRate`, `sessions`
2. **Daily trend** — date × `purchaseRevenue` / `transactions` / `sessions`

Response shape:
```json
{
  "revenue": 12500.50,
  "orders": 47,
  "aov": 265.97,
  "conversion_rate": 3.2,
  "sessions": 1470,
  "trend": [{ "date": "20260225", "revenue": 450, "orders": 3, "sessions": 120 }]
}
```

---

### Products — `GET /admin/custom/ga4/products?days=N`

**File:** `src/api/admin/custom/ga4/products/route.ts`

Two parallel reports:
1. **Per-item** — `itemName`, `itemId`, `itemRevenue`, `itemsPurchased`, `itemsViewed`, `itemsAddedToCart`, `itemRefundAmount`
2. **Totals summary** — aggregate across all products

Derived metrics calculated server-side:
- `cart_to_detail` = `(items_added_to_cart / items_viewed) * 100`
- `buy_to_detail` = `(items_purchased / items_viewed) * 100`

---

### Acquisition — `GET /admin/custom/ga4/acquisition?days=N`

**File:** `src/api/admin/custom/ga4/acquisition/route.ts`

Three parallel reports:
1. **Sources** — `sessionSourceMedium` × sessions / users / bounceRate / avgDuration
2. **Channels** — `sessionDefaultChannelGroup` × sessions / revenue / transactions / conversionRate
3. **Campaigns** — `sessionCampaignName` × sessions / revenue / transactions / users; `(not set)` filtered out

---

### Funnel — `GET /admin/custom/ga4/funnel?days=N`

**File:** `src/api/admin/custom/ga4/funnel/route.ts`

Two parallel reports:
1. **Event counts** — `eventName` filtered to `["view_item","add_to_cart","begin_checkout","purchase"]`
2. **Daily breakdown** — same 4 events by day

Derived metrics:
- `drop_off` per step = `(prev - current) / prev * 100`
- `retained` per step = `current / prev * 100`
- `abandonment_rate` = `(add_to_cart - purchase) / add_to_cart * 100`
